# Regtest e2e rig

A self-contained **regtest test rig** for the Inbound Liquidity Manager plugin.
It stands up a full local stack and installs the plugin into it, so the plugin's
behaviour can be exercised end-to-end against a *real* Electrum — the one thing
the mocked unit tests in [`../tests/`](../tests) cannot prove.

## What it launches

- **bitcoind** (regtest, RAM-capped) mining on an interval; a `rigminer` wallet
  funds each Electrum wallet.
- **Fulcrum** ElectrumX server (requires bitcoind `-txindex=1`).
- **nostr-rs-relay** in Docker (fixed container `electrum_liqtest_nostr`).
- Electrum **client** wallet with the plugin symlink-installed, and a headless
  Electrum **swap-partner** running the `swapserver` plugin (advertises LN→on-chain
  swaps at 0.5% over nostr).
- Two 0.02 BTC channels client→partner (50/50), and a local **LNURL-pay stub** so
  the dev-fee payout path can be exercised with real invoices minted by the partner.

The plugin is loaded as an **internal** Electrum plugin (auto-authorized) by
symlinking `../inbound_liquidity` into the Electrum checkout — see
`rig/services.ensure_plugin_installed()`.

## Prerequisites

`bitcoind` + `bitcoin-cli`, `docker`, `git`, `python3` on `PATH`, and the
**Fulcrum** binary at `bin/Fulcrum` (not committed — supply it out of band; it's
large). `setup.sh` checks all of these.

## Setup (one-time)

```bash
cd e2e
./setup.sh          # clones Electrum 4.7.2 into electrum/, builds .venv-electrum/,
                    # pulls the nostr image, confirms bin/Fulcrum
```

`electrum/`, `.venv-electrum/`, `bin/`, and `.run/` are **git-ignored** — they are
regenerated by `setup.sh` (in the development workspace they are symlinks to a
shared build).

## Run the rig manually

```bash
python run.py                     # brings the stack up, opens the Qt client GUI
python run.py --no-gui            # headless (both daemons; no GUI)
python run.py --exit-when-ready   # smoke: bring up, confirm readiness, tear down
```

Endpoints/nodeids/channels/swap-npub are written to `.run/ready.json`.

## Run the e2e suite

The behavioural e2e tests are gated behind `RUN_RIG_E2E=1` (each builds its own
function-scoped rig, wipes `.run`, and kills any prior rig — so do **not** run
them while a manual `run.py` rig is up):

```bash
RUN_RIG_E2E=1 .venv-electrum/bin/python -m pytest tests -q -s
```

| Test | Asserts (against the live daemon + real `wallet.db`) |
|---|---|
| `test_toggle_e2e` | the master `automation_enabled` slider gates all automation |
| `test_daily_caps_e2e` | the daily open/close ceilings hold and resume |
| `test_offline_autoclose_e2e` | only plugin-opened channels are force-closed when the peer goes offline |
| `test_manual_close_not_faulted_e2e` | a user-initiated close is not charged to the peer |
| `test_reverse_swap_e2e` | the plugin executes a real reverse swap, **increasing inbound liquidity**, and accrues a dev fee |
| `test_rig_unit`, `test_lnurl_stub` | fast rig-plumbing checks (no services launched) |

> On the rig's small (0.02 BTC) channels a reverse swap's effective all-in cost
> is several percent (a fixed ~45k-sat prepayment dominates), so at the default
> `max_swap_fee_pct = 0.6` the plugin correctly declines. `test_reverse_swap_e2e`
> raises the ceiling so a swap actually executes.

## CI note

The standard CI job runs only the fast `../tests` suite. This e2e suite needs
bitcoind + Fulcrum + Docker and takes several minutes per test, so it is **not**
part of the default pull-request CI — run it locally with `RUN_RIG_E2E=1`.
